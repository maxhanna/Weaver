# Weaver Coding Pipeline: Analysis & Improvement Roadmap

## 1. Current Pipeline Overview (UnifiedPipeline)
The `UnifiedPipeline` is the core agentic engine of Weaver. It currently follows a 5-stage process:
1.  **Bootstrap Discovery**: Project-wide scanning and initial file reads.
2.  **Context Review**: Human-in-the-loop gate for context pruning.
3.  **Planning Convergence**: Iterative JSON plan generation.
4.  **Surgical Execution**: Per-step file exploration, AST/Text resolution, and programmatic verification.
5.  **Post-Execution QA**: LLM-based verification of the final state.

---

## 2. Identified Weaknesses

### A. Context Fragmentation
*   **Issue**: Large files are truncated using `ExtractRelevantExcerpt`. This often hides class members or dependencies located in the middle of a file, leading the agent to "hallucinate" new properties instead of using existing ones.
*   **Impact**: High "edit failure" rate on complex UI components and service classes.

### B. Heuristic-Based Routing
*   **Issue**: Routing between terminal commands and code edits relies on `AgentUtilities.ClassifyTask` (regex scoring).
*   **Impact**: Ambiguous prompts often route to the wrong pipeline, wasting LLM tokens and time.

### C. Fragile AST Simulation
*   **Issue**: Only C# uses a real AST (Roslyn). TS/JS/Java/Go use regex-based brace matching to simulate `FORMAT C`.
*   **Impact**: Closing brace errors and indentation corruption in non-C# languages.

### D. Verification "Optimism"
*   **Issue**: `PostExecuteVerify` relies on LLM opinion rather than hard compiler/linter feedback.
*   **Impact**: Submitting code that "looks" right but fails to build.

---

## 3. Improvement Roadmap

### Phase 1: Context & Intelligence (High Priority)
- [x] **Skeleton File Context**: Implement a system that provides a full "map" (signatures only) for large files, expanding only the target method body.
- [x] **LLM-Assisted Routing**: Use a small, fast model to break ties in routing decisions when regex scores are close.
- [x] **Compiler-First Verification**: Integrate `dotnet build` and `npm run lint` results directly into the `PostExecuteVerify` stage.

### Phase 2: Structural Integrity
- [ ] **Tree-sitter Integration**: Move non-C# `FORMAT C` resolution from regex to Tree-sitter for proper AST support across TS, JS, and Python.
- [ ] **Multi-Edit Transaction**: Support batching multiple small edits into a single file write to reduce I/O overhead.

### Phase 3: Advanced Pipeline Optimization
- [ ] **Parallel Dependency Graph**: Analyze the plan for independent steps (e.g., creating a DTO and a CSS file) and execute them concurrently.
- [ ] **Streaming Execution**: Begin executing early steps of a plan while later steps are still being generated by the LLM.

---

## 4. Next Steps
1.  **Monitor Skeleton Efficacy**: Check if the agent correctly uses existing class members in large files.
2.  **Tree-sitter Prototype**: Evaluate the feasibility of embedding Tree-sitter for multi-language AST support.
3.  **Benchmarking**: Compare current single-threaded execution against a mocked parallel execution model.
